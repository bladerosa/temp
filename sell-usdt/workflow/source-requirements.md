## 0. 模块背景

**Sell USDT 申请** 是 CCPayment 运营后台中、专门处理 **商户用户用 USDT 兑换法币** 这一业务的运营操作面板。运营人员（操作员）在这里完成：

- 审核新提交的兑换申请
- 跟踪每笔订单从「待审核」直到「已完成」（或「已拒绝」）的全生命周期
- 在订单流程中触发与供应商的资金往来（Cwallet 转账、法币付款凭证录入、退款）
- 设置全局服务费率、查看每笔订单的核算明细

业务方向：**用户 → 平台 → 供应商**：用户用 USDT 通过平台兑换法币，平台把 USDT 中转给供应商，供应商按约定汇率把法币打给用户的收款账户。

模块入口在运营后台左侧侧边栏 `Sell USDT 申请` 项。

## 1. 全局信息（页面顶部）

### 1.1 标题与账户信息条

页面标题：`Sell USDT 申请`

标题右侧顺序排列以下信息条（自左向右）：

| # | 字段 | 取值 | 说明 |
|---|------|------|------|
| 1 | Cwallet 运营账户 ID | 例：9527321 | 整个 CCPayment 平台用于中转 USDT 的「自己的」Cwallet 账户。所有用户 USDT 流入此账户后再分发 |
| 2 | Cwallet 供应商账户 ID | 例：34575837 | 平台与供应商共用的 Cwallet 账户，平台把 USDT 转入这里，供应商即可见到 |
| 3 | 供应商汇率加点 | 例：0.5% | 平台从市场汇率中让利给供应商的百分比；体现在每一行订单的「供应商汇率」与「得到资产」计算上 |
| 4 | 「服务费设置」按钮 | 主色蓝 | 点开弹窗设置 服务费率（详见 F-服务费设置） |

账号 ID 使用等宽字体。供应商汇率加点 % 是当前生效值（来自服务费设置弹窗最近一次保存）。

### 1.2 五个 Tab（自左向右固定顺序）

| Tab | 含义 |
|-----|------|
| 待审核 | 用户刚提交、尚未操作员审核的订单 |
| 待转账给供应商 | 已审核通过，但平台还没把 USDT 转入 Cwallet 运营账户 → 系统状态为「转账中 / 转账失败 / 已完成」 |

## 6. 价格 / 费用计算（单一公式来源）

> 本节是所有列表、详情弹窗、模拟器统一沿用的公式。同一个 derive 函数：

```
平台服务费   = sellAmt × 平台服务费率
供应商承兑量 = sellAmt − 平台服务费
供应商承兑汇率 = 市场汇率 × (1 − 供应商汇率加点)
得到资产    = 供应商承兑量 × 供应商承兑汇率
外显服务费   = sellAmt − 得到资产 ÷ 市场汇率
```

「外显服务费」是把法币结果按市场汇率反推回 USDT 后与 sellAmt 的差额。这样可以把「供应商加点」的隐性损失也体现进去（默认费率下：外显 > 平台服务费）。

**默认费率验算**：sellAmt = 50,000 USDT，平台服务费率 3%，供应商汇率加点 0.5%
- 平台服务费 = 50,000 × 3% = 1,500 USDT
- 供应商承兑量 = 48,500 USDT
- 供应商承兑汇率 = 1 × (1 − 0.5%) = 0.995 USD/USDT
- 得到资产 = 48,500 × 0.995 = 48,257.5 USD
- 外显服务费 = 50,000 − 48,257.5/1 = **1,742.5 USDT**
- 外显服务费率 = 1,742.5 / 50,000 × 100% = **3.485%**

公式必须保持同源，任何调整需同步更新列表、详情、计算模拟。

## 7. 服务费设置 弹窗

入口：顶部账户信息条右侧「服务费设置」按钮。

### 7.1 弹窗结构

- 标题：服务费设置
- 唯一必填字段：**供应商汇率加点** *（百分号后缀，0–100 范围，最多 2 位小数）
- 「计算模拟」卡片置于供应商汇率加点字段之下

### 7.2 计算模拟卡片

卡片头部（同一行）：
- 左：左侧蓝色短条 + 文字「计算模拟」
- 右：**模拟平台服务费率** 内嵌输入框（96×28，带 % 后缀，placeholder「选填」）—— 这是 选填项。本字段不会写回任何全局状态，仅用于本次会话的"假设性"模拟。

卡片正文：
```
用户 sell 50,000 USDT to USD
─── 虚线分隔 ───
外显服务费   1,742.5 USDT
平台服务费    1,500 USDT
供应商承兑数量 48,500 USDT
供应商承兑汇率 1 USDT ≈ 0.995 USD
─── 虚线分隔 ───（外显服务费率 + 用户实际到手 高亮蓝色加粗组）
外显服务费率 3.485%
用户实际到手 48,257.5 USD
```

### 7.3 校验与保存

- 供应商汇率加点：必填；空 → 提示「请输入数值」；超出 0–100 → 提示「范围为 0–100」
- 模拟平台服务费率：可空，空时视为 0%；与供应商汇率加点共用 0–100、≤2 位小数规则
- 「保存」仅校验供应商汇率加点；保存后回写到顶部信息条「供应商汇率加点：x%」

## 10. 供应商退款 弹窗（仅待供应商付款 Tab 触发）

入口：待供应商付款 Tab 行操作组的「供应商退款」红色按钮。

### 10.1 业务背景

当订单进入待供应商付款后，万一发现订单信息错（如收款行错）或供应商无法承兑，需要把已转入 Cwallet 运营账户的 USDT 退回。流程：
- 进入待供应商付款状态时，系统为本订单分配 **专属退款地址池**（涵盖平台支持的 USDT 各网络），并打开监测
- 操作员把退款地址给供应商，供应商在任一支持的网络上把 USDT 转回该地址
- 操作员把链上入账流水（TxID）绑定到本订单
- 当 **绑定流水总额 ≥ 需收款数量**（即本订单的 Cwallet 数量），订单视为退款完成，自动流转为「已拒绝」
- 退款地址池与监测在 订单变成 已完成 或 退款完成 任一发生时关闭
- 若已完成后又触发退款完成事件，订单状态被改为「已取消」附原因「系统异常，请重试」（注：本期前端不实现此异常处理 UI，仅声明业务规则）

### 10.2 弹窗结构

- 标题：供应商退款地址
- 宽度 560，可滚动
- 四块卡片依次：

#### 10.2.1 拒绝原因

- 标题：拒绝原因
- 5 个单选项（前 4 个固定 + 1 个其他，其他选中时启用自由输入框）：
  1. 供应商无法继续承兑
  2. 收款信息有误
  3. 风控拦截
  4. 系统对账异常
  5. 其他 + 自由输入框（placeholder「请输入其他原因」）
- 选择「其他」之外的任一项时，自由输入框 disabled 灰色

#### 10.2.2 退款地址

- 标题：退款地址
- 网络切换 Pill 组（7 个）：TRC20 (Tron) / ERC20 (Ethereum) / BEP20 (BSC) / Polygon / Solana / Arbitrum / Optimism
- 选中网络高亮蓝
- 网络下方左侧 144×144 二维码（白底有内边距，二维码内容=当前网络的退款地址），右侧上方写「<网络名> 退款地址」12/灰，下方一行灰底胶囊显示完整地址 + 右侧复制图标按钮，再下方一行小灰说明：
  > 请使用 <网络名> 网络发送 USDT 到上述地址，单笔/多笔到账金额满足需收款数量即视为退款完成。
- 退款地址为按 (订单 recordId, 网络) 确定性合成（同一订单的同一网络始终是同一地址）

#### 10.2.3 热钱包入账流水绑定

- 卡片头：左侧标题「热钱包入账流水绑定」，**右侧蓝色文字链接「打开热钱包入账列表」**，点击在浏览器新 tab 打开 `/dashboard/hot-wallet/assets`
- 已完成绑定（=绑定总额 ≥ 需收款数量）时，卡片顶部出现黄色信息条：
  > ⚠ 已绑定金额已满足需收款数量，订单将流转至 已拒绝 状态。
- 输入行：TxID 输入框 + 「提交」按钮（主色）
- 输入框 placeholder：输入热钱包入账 TxID
- 已完成绑定时：输入框 + 提交按钮 disabled
- 绑定列表：每条绑定显示一行（mono 字体 TxID + 右侧金额 + 「解绑」按钮）；空态显示「暂无绑定流水」
- 顶部信息「已绑定 N 笔流水」
- 解绑：始终允许（即使已完成绑定状态下也可解绑，解绑后总额可能跌破阈值，输入区重新启用）

#### 10.2.4 收款进度

- 标题：收款进度
- 字段：
  - 需收款数量 = 本订单的 Cwallet 数量
  - 供应商已支付数量 = 绑定列表金额求和；达到阈值时显示成功绿

### 10.3 TxID 绑定校验（按顺序）

1. **找不到**：TxID 不在「热钱包入账流水」内 → 错误「未在热钱包入账流水中找到该 TxID」
2. **非入账**：TxID 是出账流水 → 错误「该 TxID 不是入账流水」
3. **非 USDT**：TxID 是 TRX/ETH 等非 USDT 入账 → 错误「该 TxID 不是 USDT 交易，无法绑定」
4. **已绑定到其他订单**：TxID 已被作为别的订单的供应商退款 → 错误「该 TxID 已绑定到其他订单」
5. **已被标记为其他类型**：TxID 在热钱包入账列表里已被标记为非「供应商退款」的类型（如「借入」）→ 错误「该 TxID 已被标记了，无法绑定」
6. **本订单已绑定**：本订单 bound 列表已有此 TxID → 错误「该 TxID 已绑定」

通过所有校验后：
- 在热钱包入账列表中，把该流水的 **交易类型** 改为「供应商退款」，**备注** 写入本订单 Record ID
- 本订单 bound 列表追加 `{ TxID, 该流水的真实金额 }`
- 清空 TxID 输入框

### 10.4 TxID 容错查找

操作员从列表抄写 TxID 时，看到的是截断显示形式（前 10 字符 + … + 后 8 字符）。绑定查找按以下三级容错：

1. 精确字符串相等
2. 输入 ≥ 12 字符：前 10 字符前缀匹配唯一记录
3. 输入 ≥ 16 字符：前 8 + 后 8 同时匹配唯一记录

匹中即视为找到该完整流水。模态内 bound 列表用流水的 canonical 完整 TxID 入列（不是用户输入），确保解绑能找回。

### 10.5 解绑

把该流水在热钱包入账列表的 **交易类型** 改回空（=「点击标记」），**备注** 清空。
本订单 bound 列表移除此条。

### 10.6 重打开

同一订单的供应商退款弹窗重开时，从热钱包入账列表中按 备注=本订单 Record ID 重建 bound 列表，确保跨次会话也能保留绑定状态。

## 17.bis 新增需求（v1.1）：确认付款 后 等待用户确认 过渡态

需求来源：在已有「待供应商付款」流程之上追加。

### 业务规则

1. 操作员在 待供应商付款 行点「确认付款」并把 供应商法币履约凭证 弹窗的全部内容提交成功后，**该行的 确认付款 按钮立即隐藏**，订单仍然停留在 待供应商付款 Tab（**不进入 已完成**）。
2. 该行的 详情 按钮左侧多出一行状态文字：**「等待用户确认，或 3 天后自动确认」**（小灰字，不可点击）。
3. 期间「供应商退款」按钮依然可用 —— 如发生异常仍允许走退款流程。
4. 进入此过渡态后，订单等待用户在客户端确认收款，或 3 天后由系统自动确认；任一动作完成后订单流转为 已完成。本期前端不实现「3 天后自动确认」的倒计时显示，仅展示固定文案。

### 行为约束

- 同一行多次重启会话也应保持 过渡态（按订单的系统后台状态决定，本界面只是镜像）
- 状态文字字号 12.5/grey-600，与其他行的灰字保持一致
- 状态文字内容不允许换行

### 自动确认超时与倒计时（v1.2 新增）

1. **X 取系统后台写死配置**：自动确认的截止时间 = 操作员提交「确认付款」时刻 + X 天；当前 X = 3。后台配置变更时本界面跟随，不单独维护此常量。
2. **单位自动降级**：状态文字按剩余时间动态选用「天 / 小时 / 分钟」单位：
   - 剩余 ≥ 1 天 → 显示「N 天后自动确认」
   - 剩余 < 1 天且 ≥ 1 小时 → 自动降级显示「N 小时后自动确认」
   - 剩余 < 1 小时 → 自动降级显示「N 分钟后自动确认」（下限 1，避免出现「0 分钟」）
3. **每次刷新页面（或切回 Tab）即重算剩余时间**：状态文字按「截止时间 − 当前时刻」即时计算，不做周期性实时倒计时滚动。
4. **示例**：「等待用户确认，或 3 天后自动确认」/「等待用户确认，或 12 小时后自动确认」/「等待用户确认，或 5 分钟后自动确认」
